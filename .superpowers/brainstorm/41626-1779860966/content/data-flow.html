Section 5 — Data Flow & Auth

How the Next.js dashboard talks to Spring Boot and Keycloak

Authentication Flow

Browser → NextAuth.js → Keycloak
1. User visits /admin → middleware checks session → redirects to Keycloak login
2. Keycloak authenticates, returns auth code → NextAuth exchanges for tokens
3. JWT stored in httpOnly session cookie. Role + email claims extracted.
4. Middleware reads role from token → gates /admin/* routes

Data Fetching Patterns

Server Components (RSC)
Pages that don't need interactivity fetch data on the server:
// app/(admin)/dashboard/page.tsx
export default async function DashboardPage() {
const stats = await fetch(
`${API_URL}/v1/admin/stats`,
{ headers: { Authorization: `Bearer ${token}` } }
).then(r => r.json());
return <DashboardStats data={stats} />;
}
Best for: Dashboard, Lots list, Audit log
Client Components + SWR
Interactive pages (search, filter, mutate) use client-side fetching:
// app/(admin)/(superadmin)/users/page.tsx
'use client';
export default function UsersPage() {
const { data, mutate } = useSWR(
`/api/admin/users?role=${filter}`,
fetcher
);
// mutate() after PATCH role change
}
Best for: Users table, Violations queue, Settings form

Mutation Flow (e.g., changing a user's role)

Click role dropdown → ConfirmDialog → fetch PATCH → Spring Boot validates + persists → SWR revalidates → Toast: "Role updated"
Backend writes audit_log entry. UI optimistically updates, reverts on error.

Error & Loading States

Loading Glass skeleton cards with shimmer animation (matches Android app)
Empty Centered illustration + "No violations found" copy with clear filters CTA
Error Toast notification with retry. Inline error for table rows that fail individually.
Unauthorized Middleware redirects to /login. API 403 shows "insufficient permissions" banner.
